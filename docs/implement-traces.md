- **Alloy** runs `otelcol` (OpenTelemetry Collector) вҖ” it receives OTLP traces from apps, batches them, and forwards to Tempo.
- The OTel SDK inside each app buffers spans in memory and exports them every ~5s or when the buffer fills up. It's push-based вҖ” the app initiates the gRPC connection to Alloy.

## How trace context propagates

- A W3C `traceparent` header (containing `trace_id` + `span_id`) is generated by the Python producer on `POST /ingest` and injected into RabbitMQ AMQP message headers.
- The Go consumer reads `traceparent` from the AMQP headers, restores the trace context, and creates a child span вҖ” same `trace_id`, new `span_id`.
- Every log line should also include `trace_id`/`span_id` so Grafana can correlate logs with traces.

## What you need

| Component | Role |
|-----------|------|
| Python producer | OTel SDK + Flask auto-instrumentation. Pushes spans to Alloy. |
| Go consumer | OTel SDK. Extracts context from AMQP. Pushes spans to Alloy. |
| Alloy | OpenTelemetry Collector. Runs `otelcol.receiver.otlp` вҶ’ `otelcol.processor.batch` вҶ’ `otelcol.exporter.otlp`. Forwards to Tempo. |
| Tempo | Push-based trace storage backend. Receives OTLP on `:4317`. |
| Grafana | Queries Tempo datasource for trace exploration and Drilldown views. |
